# 2026-04-07 工作记录

## 自动化更新失败诊断与修复

### 问题描述
自动化任务最近（2026-04-04 到 2026-04-07 连续4天）未运行成功，导致数据停留在 2026-04-03。

### 根本原因（两个独立 Bug）

**Bug 1：who-is-hiring 解析 NoneType 错误**
- `work_mode`/`company`/`salary` 等字段在 Rebase API 中可能返回 `null`
- `"远程" not in None` → TypeError，导致整个 who-is-hiring 段解析失败
- `requirements` 字段类型不一致（590/787 条为字符串，197/787 条为 null，0 条为数组）
- `contact_channels` 为 None 时 `', '.join(None)` 报错
- 导致 money.xlsx 写入也失败

**Bug 2：jobs-global.js 超过 GitHub 100MB 限制（347MB）**
- 远程岛 API description 字段含完整 HTML，单条最大 3.4MB
- 虽然代码有 `[:500]` 截断，但旧数据中 968 条已写入了超长 description
- 历史数据不断 merge 进来，文件越来越大
- GitHub pre-receive hook 拒绝推送（超过 100MB 限制）

### 修复方案
1. `scrape.sh` 中对 who-is-hiring 所有可能为 null 的字段加兜底（`or ""`/`or []`）
2. `requirements` 兼容 string/list/null 三种类型
3. `build_job_desc()` 中处理 None 的迭代
4. Remote OK description 加 `[:500]` 截断（之前漏了）
5. 用 Node.js 重新生成 jobs-global.js，对全部 description 截断到 500 字符（347MB→2.5MB）
6. `git reset --soft` 合并大文件提交，重新推送成功

### 修复后数据
- who-is-hiring：成功解析 434 条
- CN：537 条，Global：1505 条
- money.xlsx：成功插入 434 条 Rebase 数据
- Git 提交 `792b788` 已推送 master
