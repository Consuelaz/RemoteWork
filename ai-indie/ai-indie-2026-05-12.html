AI 独立派

AI 让工作更复杂了——这恰恰是独立开发者的好消息

本文由 AI 辅助整理，观点源于独立开发实战与真实 builders 的一手声音。

"AI will naturally cause most roles to actually grow in complexity rather than reduce in complexity, because we can do far more with the tools."

这是 Box CEO Aaron Levie 上周发在 X 上的一段话。不是反 AI，不是焦虑帖，是他认真思考之后的判断。

很多人听到这话第一反应是不解——AI 不是应该让工作变简单吗？但细想一下，你就会发现 Levie 说的是另一件事：AI 让每个人的天花板升高了，所以大家都开始做更难的事。

这背后藏着一个对独立开发者至关重要的认知重置：如果你还在用 AI 做"节省时间"的算术题，你可能早就算落后了。

01

AI 工具不是减速带，而是赛道加宽了

Levie 的逻辑链是这样的：每个人都能用 AI 加速执行，意味着"基础任务"的门槛崩塌了。以前写一份合同初稿要一周，现在 AI 一小时搞定——但这不代表律师失业，而是意味着律师开始做以前没时间做的"深度法律分析"。

工程师亦然。他说得很直白："能用 AI 的工程师，会比不懂技术的人用 AI 写同样软件，产出高得多。"

在过去一年，"谁都能用 AI 写代码"的叙事让不少人开始怀疑：独立开发者的技术壁垒还有价值吗？但 Levie 给出了另一个角度——技术能力不是被抹平，而是被放大了。你懂架构，AI 帮你快速搭；你懂业务，AI 帮你快速验；你懂用户，AI 帮你快速迭代。非技术创业者用 AI 做出的 MVP 往往只是功能堆砌，而你做出来的是有深度的东西。

这种放大效应，在 OpenAI 内部已经有了真实的证据。OpenAI 首席科学家 Ako Paioki 在播客里透露，他们现在用 Codex 做大量实际的科研任务——不只是写代码，而是运行在更长时间窗口里的自主研究任务。他说了一句让人印象深刻的话：模型已经在解一道他博士期间研究过的数学难题，"看到它在一小时内想出我可能需要一两周才能想出的思路，真的是很奇怪的感觉"。

这不是噱头。这是 AI 工具真正进入"智力合作者"阶段的现实信号。对独立开发者来说，这件事的底层影响是：你需要重新定义什么是"你的工作"——不是写代码，而是知道该写什么。

02

先聊用户，再让 Agent 做执行——新工作流已经跑通了

Zara Zhang（独立 builder）上周发了一条推文，谈到她观察到的新工作方式：产品团队应该把更多时间花在外部沟通（跟用户、客户）上，而不是内部协作。

原因她说得很清楚：AI 很擅长执行，但问题定义只有人才能做；团队变小了，内部协调的开销正在缩小；已经有独立 builder 在跑一套新工作流：整天跟用户聊，录音丢给 Agent，Agent 判断该做什么、负责执行。

但这里有一个关键的认知陷阱值得拆解："让 Agent 做执行"不等于"放手不管"。

Ako Paioki 明确说，当前模型即使是 Codex 这种级别，也还处于"需要给定具体任务的研究实习生"阶段，而不是"能自主制定研究方向的研究员"。他预计还要到某个节点，模型才能真正"自主工作几天并产出高质量成果"。

现阶段最有效的工作切分：

  • 你负责：用户访谈 → 问题定义 → 功能优先级判断 → 最终产品方向
  • Agent 负责：需求拆解 → 代码实现 → 测试 → 文档 → 重复性内容产出

这不是激进前沿玩法，是 2026 年已经在跑通的普通工作方式。如果你还在把 AI 用来"写一封邮件草稿"，你已经落在了曲线后半段。

03

你的产品安全吗？Vercel 事件的另一个教训

Vercel CEO Guillermo Rauch 上周公开了一起安全事件：一名员工的账号被黑，攻击者借此进入了 Vercel 内部环境，访问了部分客户的环境变量。

Rauch 在声明里说了一句很多人忽略了的话："我强烈怀疑攻击者大量使用了 AI，他们的速度和对 Vercel 系统的理解程度令人惊讶。"

这是一个在网络安全领域已经被反复提及但还没被充分重视的现实：黑产也在用 AI，而且可能比你的安全团队用得更激进。

"人类将持续是攻击的主要向量，而且我们还什么都没看到。"—— Nikunj Kothari（FPV Ventures 合伙人）

对独立开发者来说，这件事的反直觉之处在于：AI 时代，安全工作的优先级不应该降低，而应该升高——因为你的代码产出速度更快了，意味着你把代码推向生产的速度也更快，安全审查的窗口更短了。

04

这几天 builders 圈值得关注的事

🛠 Garry Tan 的 GStack v1.4

YC 总裁 Garry Tan 发布了 GStack v1.4，新增 /make-pdf skill。他的 workflow：遇到需求就让 Claude Code 实现，然后开源。

对独立开发者的价值不在于工具本身，而在于"自生长工具链"的思维：Agent 工作流里缺什么，就用 Agent 造什么，然后开源给社区用——低成本获取早期用户的有效途径。

🤖 "该不该做自己的强化学习？"OpenAI 首席科学家的回答

Ako Paioki 的务实建议：比起复制 RL pipeline，把领域知识喂进上下文（in-context learning）可能是更高效的路径——而且未来模型会越来越擅长这件事。高质量的用户对话、业务案例、决策记录，整理成上下文模板，ROI 远高于自定义 RL。

⚠️ Peter Yang 的 OpenClaw 切换模型踩坑

Peter Yang 把 OpenClaw 切到 GPT 模型，连简单的 cron job 都搞不定。不同模型的 agentic 执行能力差异很大，Opus 表现明显更好。

实用提示：不要因为省钱就随意换底层模型，先在关键 workflow 上做基准测试再迁移。

05

如果你今天只做一件事

行动 1：把 AI 用在"问题定义"上，不只是"执行"

去录一段真实的用户反馈（哪怕是一条差评、一封邮件），把它完整地丢给 AI，让 AI 总结核心痛点和功能需求，然后你来判断方向。10 分钟，但形成习惯之后，产品决策质量会有质的提升。

行动 2：做一次 API Key 安全审计

打开你最近项目的 .env 文件，问自己：哪些 key 泄露了会造成损失？多久没轮换了？有没有设置访问告警？30 分钟可以搞定，不需要安全专家。

行动 3：用 GStack 的思路改造你的 Agent 工作流

你现在最常用的 Agent 工作流里，有没有哪个操作是重复手动做的？写一个 skill 或 prompt 模板把它固定下来。如果做出来有通用价值，开源出去。

Levie 说"AI 让工作变复杂"，Ako Paioki 说"模型还差得远"，Vercel 说"黑产在用 AI 攻击"——这三件事放在一起，听起来像焦虑。

但这里有一个意外的利好：小团队和独立开发者，是目前最能灵活利用这些工具的群体。

大公司做模型选型要走审批，推 Agent 工作流要培训全员，改安全规范要等合规部门。而你，明天就可以把这篇文章里提到的三个行动全部做完。

这是一个规模越小、反应越快、收益越大的时代。别想太多，先跑起来。
