别再只做 MVP 了：AI 时代，脚手架才是你的核心竞争力

本文由 AI 辅助整理，观点源于独立开发实战与真实 builders 的一手声音。
"We used to say just do the bare minimum to get the MVP out. Don't spend time on systems. It's literally reversed now."

—— Ryan Carson（引自 Peter Yang 推文）

这是 Ryan Carson 说的。他现在每天用 AI agents 提交 10 个 PR。

这句话让我停下来想了很久。过去十几年，"先做 MVP、快速验证、别过度工程"几乎是独立开发者的宗教教条。现在 Ryan 说这个逻辑反转了。

不是"也许"反转了，是 literally reversed。

这背后到底发生了什么？对独立开发者来说，这意味着什么？

01 / 为什么"先搭系统"不再是浪费

Ryan Carson 的核心论点是：以前你花两周搭脚手架，是在浪费本该用来验证市场的精力。现在你花两小时让 AI agent 帮你搭好文档、测试、CI、skill file，然后这个 agent 能帮你干 10 个人干的活。

你省下的不是两小时，是接下来几个月里每周 20 小时。

Peter Yang 在推文中详细记录了和 Ryan 的对话。Ryan 现在的 workflow 是：

  1. 花大量时间把文档、规范、skill file 全部写好
  2. 把这些写成 cron job + skill file
  3. 然后突然间，你一个人的产出等于 10 个人

这与传统 MVP 思维完全相反——过去先搭系统是被嘲笑的"拖延症"，现在不搭系统才是真正的短视。

对独立开发者的底层影响是：你的竞争力不再来自于"谁先上线"，而来自于"谁的系统能让 AI 持续帮你产出高质量代码"。MVP 思维让你跑得快，但 AI 时代的胜利属于能持续跑的人。

02 / 独立开发者最需要重新理解的：你和 AI 的协作边界

这个反转不只是"多写点文档"这么简单。它实际上在重新定义独立开发者的核心能力。

过去（MVP 时代）

你的价值 = 你能写多少代码 × 你的产品直觉

现在（AI 协作时代）

你的价值 = 你能把多少上下文喂给 AI × 你的系统能跑多稳

这个转变有三个具体场景，每个独立开发者都该重新思考：

场景一：技术选型
以前你选轻量框架，因为你要自己扛所有代码。现在你选什么？选AI 最熟悉、文档最全、社区最大的框架。因为你的 agent 会用它。你需要的是让 AI 写出高质量代码，而不是展示你的小众品味。

场景二：代码质量
Nikunj Kothari（FPV Ventures 合伙人）说了一句很犀利的话："Automate or get automated is the bitter lesson we all need to learn." 他是一个 VC，但他每周都在 build——因为"这个领域变化太快了，唯一能站在前沿的方式就是亲手 build"。

如果你还在用 2023 年的方式写代码（手写每个功能、手工测试、手动部署），你不是在怀旧，你是在被淘汰。

场景三：依赖管理
Peter Steinberger（OpenClaw 作者）展示了一个具体案例：他把 OpenClaw 的依赖从 Sharp + Jimp（140MB）替换成了 photon（2MB，Rust 编译的 WebAssembly）。

140MB → 2MB。

这不是洁癖，这是认知。当你的 agent 需要频繁启动沙箱环境来运行代码时，依赖体积直接影响响应速度和成本。Steinberger 的另一个发现更值得注意：他写了一个 skill，专门找出那些在 skill description 里写小说的 offender——因为那些废话每次都被加载进 context，白白消耗 token。

对独立开发者的行动建议：今天回去审查你的项目——依赖体积大吗？README 和文档足够让 AI 理解你的项目吗？你的 skill files 写得够精简吗？这三点，决定了你的 AI 协作效率。

03 / AI 不会消灭工作，它会重新定义"足够好"

Aaron Levie（Box CEO）转了一条 Goldman Sachs CEO 的观点，很有意思。

Goldman Sachs CEO 站在"AI 乐观派"一边，反对那些"AI 会导致大规模失业"的悲观论调。他的逻辑是：

历史上每次自动化都让我们"要求更多"，而不是"做得更少"。
我们获得了更全面的金融分析，更深度的法律建议，更定制化的金融服务。

这个观点对独立开发者特别重要，因为很多人现在焦虑："AI 这么强，我还能做什么？"

Levie 的回答是：市场会不断抬高预期。当所有人都能用 AI 快速生成一个 SaaS，用户不会说"哇你好厉害"，他们会说"这个功能怎么还没有？那个集成怎么还不支持？"

AI 降低了"做出东西"的门槛，但抬升了"用户期待"的天花板。

所以你的机会不在"用 AI 做一个竞品"，而在：用 AI 做到那些大公司因为流程臃肿而做不了的事情——这就是独立开发者的意外优势。你一个人 + 10 个 agents，决策速度为 0，没有会议，没有审批。这是大公司永远做不到的。

04 / 这几天 builders 圈值得关注的事

Daytona 与 AI Agent 沙箱基础设施

Daytona CEO Ivan Burazin 在 MAD Podcast 上有一个非常硬核的分享：为什么 AWS 和 Azure 无法运行真正的自主 AI agent。核心论点是：每个 agent 都需要自己的沙箱计算机，而不能共享一个容器。Daytona 因此抛弃了 Kubernetes，重写了自己的调度器。

对独立开发者的价值点：如果你正在做 AI agent 产品，沙箱隔离是你迟早要面对的架构问题。Daytona 的开源方案值得提前了解。

Codex vs Claude：前端任务仍然属于 Claude

Peter Yang 实测了两个顶级编码 agent：Codex 很强，尤其是它用浏览器自己测试自己的代码这个能力非常惊艳。但是——任何涉及设计或前端的任务，Claude 仍然碾压。

建议：后端逻辑交给 Codex，前端精细调整交给 Claude。两个都用，不要二选一。

Garry Tan 的 Evals 工作流

Y Combinator CEO Garry Tan 分享了一个具体方法：让你 agent 用三个不同的前沿模型互相评估你的 skill file 和代码输入输出，问它们"为什么这不是 10 分？"跑几次效果会让你惊讶。而且因为评估逻辑写在 skill file + 单测里，改进是永久的。

05 / 如果你今天只做一件事

  1. 审查你当前最主要项目的 README 和文档。问自己：如果一个 AI agent 完全不懂我的项目，只看这些文档，它能写出符合我预期的代码吗？如果不能，现在就补。
  2. 找一个你反复手动做的任务（部署、测试、数据备份），用 AI 帮你写成脚本或 skill file。不是为了今天省时间，是为了接下来 6 个月每周都省时间。
  3. 去试用一个你没用过的 AI 编码工具（Codex、Claude、Cursor），用它重写你项目里的一个真实功能。你会发现自己之前对"AI 能做什么"的理解至少落后了三个月。
过去独立开发者最大的劣势是：资源少、人手少、跑得慢。

现在，如果你理解了 Ryan Carson 说的那句话，搭好了系统，你会发现：资源少不再是劣势，反而成了优势。
大公司要花两年才能完成的 AI 转型，你一个人两个周末就能搞定。
