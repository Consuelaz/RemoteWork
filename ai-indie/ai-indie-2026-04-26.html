工作不会变少，只会变难——AI 时代真正的稀缺是什么

"AI 自然会让大多数职位的复杂度上升，而不是降低——因为我们能做的事情更多了。"

这是 Box CEO Aaron Levie 上周说的话。他不是在安慰你，他是在描述一个冷酷的事实：当所有人都能用同一套 AI 工具，你以为自己有了超能力，结果发现跑道变长了，门槛也抬高了。

这句话和 OpenAI 首席科学家 Ako Paioki 在 Unsupervised Learning 播客里的观点几乎同频——模型会变得更自主，能跑几天的任务；但这不意味着"工程师可以消失"，恰恰相反，真正的挑战是：你能设定足够好的方向吗？

这就是我想今天和你聊的核心矛盾：AI 解放了双手，但把更重的责任压在了脑子上。

01 / 能力上升，但"够用"的标准也在上升

Aaron Levie 的完整逻辑是这样的：AI 让每个人都能做更多，所以做更多本身不再是差异化。以前写一份合同是律师助理的本职，以后那不叫工作，那叫"启动流程"。以前独立开发一个轻量 App 已经很厉害，以后那只是"发布了一个 demo"。

这听起来有点令人沮丧，但其实是一个结构性利好的信号：专业能力越深，AI 放大越大。

OpenAI 首席科学家 Ako Paioki 的话印证了这一点。他在播客里说，区分"AI 研究实习生"和"真正自主的 AI 研究员"的关键，是任务粒度和自主跨度——模型可以在你给出具体技术方向后执行，但"去解决对齐问题"这种宏观指令，不是今年能实现的事。

翻译成独立开发者的语言就是：你越懂自己的领域，你设定的任务粒度就越精确，AI 就越能为你放大产出。

一个对用户问题理解透彻的独立开发者，让 Claude Code 来实现细节；一个连问题本身都说不清楚的人，会发现 AI 不断在"误解需求"的方向上发力。AI 成了一面镜子——照出你是否真的理解自己在构建什么。

这对独立开发者的底层影响是：你的核心竞争力不再是"能写代码"，而是"能清晰描述问题、判断方向、决定优先级"。这三件事，AI 做不了——至少今年还做不了。

02 / 外部沟通变成了独立开发者的第一资产

Zara Zhang（GitHub 作者，前 AI 领域研究者）上周发了一条很值得细品的推文：

"AI 越厉害，产品团队就应该把更多时间花在外部沟通（和用户、客户聊天）上，而不是内部沟通上……我见过一些独立创业者，他们整天跟用户对话，然后把录音直接丢给 agent，让它搞清楚接下来要做什么。"

这不是隐喻，这是 2026 年独立开发者工作流的现实描述。

Ako Paioki 在访谈里也说过类似的话：模型能力的瓶颈不一定是智力，更多时候是"上下文不足"——你没有把正确的文件、正确的系统权限、正确的业务背景喂给它。而谁最了解产品的真实上下文？是和用户深度对话过的你。

对独立开发者来说，这意味着一个具体的工作流转变：

以前的工作流：想法 → 设计 → 编码 → 上线 → 找用户验证

新的工作流：找用户 → 深度对话 → 把问题和上下文整理清楚 → 丢给 AI 执行 → 快速迭代

这个转变的核心不是"少写代码"，而是把稀缺时间从实现环节，挪到理解问题环节。

差异化行动建议：

  1. 建立"用户对话记录库"：每次和用户聊完，用 30 秒语音备忘，然后让 AI 整理成结构化问题列表。这不是文档，是你的产品 Context 库。
  2. 每周做一次"需求 ↔ 实现对比"：对照你让 AI 实现的功能，和用户真正的反馈，找到 gap。Gap 越小，说明你的需求表述越精准——这是一个可以刻意练习的能力。
  3. 减少内部文档，增加外部接触：团队只有你自己时，写给自己看的文档是成本，和用户的一次对话是资产。

03 / "AI 让人变懒"是一个方向性错误的担忧

这周我看到不少独立开发者在讨论一件事：AI 工具用多了，是不是会让自己变得越来越依赖、越来越浅？

我觉得这个担忧本身就是一个框架错误。

Ako Paioki 在访谈里说了一句话，我觉得适合拿来反驳这个焦虑：

"如果你真的能做更多，你就会开始挑战更大、更难的问题。以前的工作定义就不再适用了。"

这和 Aaron Levie 的逻辑是一体两面：工作不会变少，会变难。一个用 AI 实现基本功能的开发者，不会停下来说"够了"——他会开始想"那我能不能做一个更复杂的数据 pipeline？能不能支持多语言市场？能不能加推荐算法？"

真正"变懒"的风险，不是 AI 工具用多了，而是你没有升级自己设定问题的能力。

Garry Tan 上周分享了他的工作方式：每次在使用 Claude Code 的过程中发现一个新需求，就让 Claude Code 直接做成工具开源出去。GStack v1.4 新增了 /make-pdf 功能，他没有等一个版本周期，他"随需构建、随手开源"。

这是一种极其高效的独立开发者心态——让 AI 处理实现的摩擦，把自己的注意力专门留给"这个值不值得做"的判断。

对独立开发者来说，这个反直觉洞察是：担心 AI 让你变懒，不如担心你的判断力没跟上你的执行速度。

04 / 这几天 builders 圈值得关注的事

Vercel 遭遇 AI 加速的供应链攻击

Vercel CEO Guillermo Rauch 上周发了一条严肃声明：一名员工因使用的 AI 平台 Context.ai 被攻破，攻击者借此入侵了 Vercel 的 Google Workspace 账户，进而访问了部分 Vercel 环境变量。他特别提到："我强烈怀疑，这个攻击团队被 AI 显著加速了——他们行动速度惊人，对 Vercel 内部架构的理解异常深入。"

对独立开发者的核心警示：你连接的每一个 AI 工具，都是一个潜在的攻击面。不要把生产环境的敏感变量标记为"非敏感"；定期轮换 API Key；把第三方 AI 工具当成潜在的弱点，而不是黑盒信任它。这次事件提醒我们，AI 不只是防守方的工具——攻击方也在用。

Peter Yang 实测 GPT 模型做 Agent 任务：翻车体验

Peter Yang（Roblox 产品负责人，140K 订阅作者）上周公开分享了他用 OpenClaw（Claude Code 对标工具）+GPT 模型处理 agent 任务的失败经历：他让它发一封周报邮件，结果模型把邮件模板搞乱，换了 Sonnet 还是不行。

他的结论是：这个模型在跟踪长链条 agentic 任务方面就是不行，不是用法问题，而是模型本身的问题。值得关注的是，评论区里大量 AI builder 群体也反映类似问题。

对独立开发者的启发：不要把 Agent 任务的失败完全归因于 Prompt 写法。在 GPT 和 Claude 之间做 Agent 选型时，先用小任务测试模型对"多步跟踪"的稳定性，而不是看评测榜单。选 Claude Sonnet 系做 agent 目前仍是更稳的选择。

GStack v1.4：随需构建、随手开源的独立开发者范本

Garry Tan（YC 总裁）在使用 Claude Code 过程中发现需要一个 PDF 生成工具，直接让 Claude Code 做了，顺手打包成 /make-pdf 技能发布到 GStack v1.4。他说这就是他的工作方式：遇到需求就做，做了就开源。

这条动向本身不算大新闻，但对独立开发者来说，它代表了一种值得学习的工作哲学：你的每一个"临时工具"，都可能是别人的"刚需产品"。用 AI 快速实现，用开源快速验证，用社区反馈决定是否值得继续。

05 / 如果你今天只做一件事

行动一：重新评估你的工具链安全性（15 分钟）

受 Vercel 安全事件启发，打开你所有正在使用的 AI 工具，检查：有没有连接到生产数据库或 API Key？这些 Key 最近有没有轮换过？把"方便用"的设置和"安全用"的标准对齐一次。这件事做一次，能省掉未来很多麻烦。

行动二：做一次"问题表述练习"（30 分钟）

拿出你当前最重要的一个未解决的产品问题，尝试用三句话说清楚：这个问题的背景是什么（用户是谁）、问题的核心症状是什么（用户在哪里卡住）、什么是你想要的结果。然后把这段描述丢给 Claude，看它能不能直接给出方向，不需要你补充细节。如果需要反复追问，说明你的问题表述还不够清晰——而这才是你最该投资提升的能力。

行动三：这周和一个真实用户深聊一次（60 分钟）

不是问卷，不是 NPS。是一次开放式对话，问他们最近在哪件事情上卡住了、如果有魔法棒会怎么解决。把录音整理成文字，丢给 AI 提炼 3 个核心问题。这个信息，比任何竞品分析都值钱。

反转视角：这周最让我觉得有意思的，不是任何一个新功能，而是 Ako Paioki 最后说的一句话：

"我相信技术挑战会变成'历史'，但真正的问题会变成：什么是值得做的事情？"

这个世界上最稀缺的东西，从来不是执行力。在 AI 让执行力趋于商品化之后，品味、判断力、和真正理解人的能力，会变得比任何时候都值钱。

作为独立开发者，你手上多了一批极其得力的工具——但这把锤子，只有在你知道要砸哪颗钉子的时候，才真的有用。
